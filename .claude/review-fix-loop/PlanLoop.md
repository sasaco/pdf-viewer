あなたはオーケストレーターです。
計画文書群（`docs/plan/<area>/*.md`）に対して「レビュー → 集約 → 統一方針決定 → 修正 → 検証」を 1 ラウンドとし、MEDIUM 以上の Finding がゼロになるまで反復させます。

> このスキルは PlanLoop-<area>.md（タスク概要・前提資料・対象ファイル一覧を bespoke で書いた呼び出しファイル）から `Read` で読み込まれて使われる前提です。汎用ロジックはこちら、案件固有情報は呼び出し側で。

## 0. 起動チェック

ラウンド 1 開始前に必ず以下を実行する:

1. 対象ディレクトリを `ls` し、計画文書ファイル一覧を確認
2. `review-fixes-<日付>.md`（修正ログ）の存在確認。**無ければラウンド 1 として新規作成する**。今後の日付は `currentDate` を使う
3. 対象ファイル合計行数を `wc -l` で測り、レビュー範囲の規模感を把握（4000 行超ならレビュー観点を分割追加検討）
4. 直前 1〜2 コミットの差分を `git log --oneline -5` で確認し、計画変更の背景を把握

## 1. ループ手順

### Step 1: レビュー（サブエージェント並列）

**4 観点（A/B/C/D）を 4 つの `general-purpose` サブエージェントに並列で割り当てる**。

| 観点 | 内容 |
|---|---|
| **A 文書間整合性** | 計画群（README/spec/architecture/implementation-plan/open-questions など）+ 上位 SKILL の間で矛盾・旧表記・章節リンク死活・用語統一・enum 表記揺れ・フェーズ番号混在 |
| **B 既存実装・依存計画とのズレ** | コードベース実装済み（Phase 前段）と移植元（他リポジトリ）の関数名・型・呼出規約・フィールド構成・Debug マスク方針が計画記述と食い違っていないか。依存計画（他フェーズ）との引き取り境界 |
| **C 仕様漏れ・設計リスク** | 一次資料 SKILL（R1〜RN / API 規約 / セキュリティ規約 / エンコード規約 / 採番規約）から見て計画文書で未対処・曖昧の箇所。互換不変条件（API 境界の漏出禁止語）違反 |
| **D テスト不足** | 計画に書かれた実装タスクに対して、受け入れ条件・単体・結合・E2E・回帰・lint・CI ゲート組込の観測点（実行コマンド・テストファイル名・assert 内容）が明記されていない箇所 |

**各サブエージェントへの共通指示**:

- Findings は **HIGH / MEDIUM / LOW** に分類。各件「ファイル:行範囲 / 問題内容 / 推奨修正方針」を添える
- **`review-fixes-<日付>.md` に既出の Finding は除外**（プロンプト本文で明示）
- **新規 Finding のみ報告**
- **語数上限を切る**（観点 A 400 語 / B 400 / C 400 / D 400 程度）
- 終盤ラウンド（HIGH/MEDIUM が減ってきた段階）では「**収束確認モード**: HIGH/MEDIUM ゼロなら『収束』と明記」を付ける
- 案件固有のスコープ外指摘（例: 他 venue 追加、別フェーズ巻き取り）を出さない旨を明記
- SKILL の一次資料を計画文書で上書きしない方針を明記

**並列起動は必ず単一メッセージ内の複数 `Agent` 呼出で行う**（直列起動は時間の浪費）。`run_in_background: true` 推奨。

### Step 2: 集約と統一方針決定（オーケストレーター本人）

サブエージェントの結果を読み終えたら、以下を**必ずオーケストレーター本人が決定する**:

1. **HIGH/MEDIUM/LOW を整理して件数を出力**（出力形式は §2）
2. **終了条件チェック**:
   - HIGH/MEDIUM ゼロ → ループ終了 → §3 完了サマリ
   - 残存あり → 次ステップ
3. **統一方針（先決事項）の決定**:
   - 同一 Finding の解決策が複数ファイルに横串で波及するもの（用語統一・enum 値・HTTP ステータス選択・命名規約・ガード方針 など）について、**ラウンド内で 1 つの決定に集約**する
   - 例: `RATE_LIMITED` HTTP コードを 429 にするか 503 にするか / `order_number` を `venue_order_id` に統一するか / 引数追加 vs builder pattern
   - **これらの決定はテキストでまとめて表示**し、後続の implementer サブエージェント全員に同一文言で渡す（プロンプト先頭に「統一決定」セクションを置く）

> ⚠ **ここを飛ばすと修正サブエージェントごとにバラバラの判断をして次ラウンドで再分裂する**。今回のループ運用での最大の知見。

### Step 3: 修正（サブエージェント並列、ただし**ファイル単位で分割**）

**Finding 単位ではなくファイル単位**で `implementer` サブエージェントに割り当てる。同一ファイルへの並列書き込みは Edit 競合・整合崩れを起こす。

| 修正エージェント割当 | 担当 |
|---|---|
| ① README.md 担当 | README の関連 Finding を全部集約 |
| ② spec.md 担当 | spec の関連 Finding を全部集約 |
| ③ architecture.md 担当 | architecture の関連 Finding を全部集約 |
| ④ implementation-plan.md 担当 | implementation-plan の関連 Finding を全部集約 |
| ⑤ open-questions.md + review-fixes ログ担当 | 残り + 当ラウンドのログ追記 |

**各 implementer への指示テンプレ**:

```
docs/plan/<area>/<FILE>.md を以下の Finding に従って修正する。
**<FILE>.md のみ書き換え**。他は Read のみ可。

【統一決定（全エージェント共通）】
- ＜オーケストレーターが Step 2 で決めた方針を 1 行ずつ＞

【Findings】
- **<ID> (HIGH/MEDIUM/LOW) <タイトル>**: <詳細> / <推奨修正>
- ...

修正後は対象節を Read で確認し反映を verify。
最後に「<FILE> 修正完了 / 反映節: ...」とだけ報告。
```

**ログ更新エージェント（⑤）の責務**:
- `review-fixes-<日付>.md` 末尾に「## ラウンド N（日付）」セクションを追記
- 全 Finding を表形式（Finding ID / 観点 / 対象ファイル:行 / 修正概要）で記録
- 当ラウンドで decided な統一方針も冒頭に箇条書き

### Step 4: 修正の検証（重要）

サブエージェント完了通知を受け取ったあと、**修正済みの宣言を信用しすぎない**:

- 用語統一のような横串修正の場合、**`Grep` で残存件数を機械的に確認**（例: `Grep "order_number"` で旧用語が残っていないか）
- 章節リンクのアンカー死活は、修正後の md ファイルから再度 `Grep "spec.md#"` でリンクパターンを抜き出し、対応する見出しが存在するか抜き打ち確認
- これらは費用対効果が高いので**ラウンドごとに 1〜2 回**走らせる

### Step 5: 次ラウンドへ

§Step 1 に戻る。次ラウンドのレビュー指示には:

- **当ラウンドで修正された箇所を重点検査**する旨を明記
- 統一方針（前ラウンド決定）を冒頭に再掲し、その方針が他箇所に整合的に波及しているかを観点 A に追加検査させる
- ラウンド数が増えたら観点を統合（例: 「観点 A+B+D を 1 エージェントで合算」）して並列度を下げる

## 2. 出力形式

各ラウンド開始時:

```
=== ラウンド N ===
残存 HIGH: X件 / MEDIUM: Y件 / LOW: Z件

【統一決定】
- 決定 1: ...
- 決定 2: ...
```

各 implementer 起動時はサブエージェントが「修正完了 / 反映箇所」を報告するので、オーケストレーターは要点だけ転記。

## 3. 完了サマリ

```
=== 完了 ===
全ラウンド数: N
修正した Finding 総数: HIGH X / MEDIUM Y / LOW Z
残存 LOW（対応不要）: K件
主要な反映成果（規約レベル）:
- 用語: ...
- reason_code: ...
- 不変条件: ...
- 設計規約: ...
- テスト・CI: ...
ログ: docs/plan/<area>/review-fixes-<日付>.md
```

## 4. ループ上限と escape hatch

- **最大 N ラウンド = 8** をハード上限とする。それを超えても収束しない場合は強制終了し、残存 HIGH/MEDIUM を `open-questions.md` に **未決オープン質問として書き出す**
- ラウンド数が増えても HIGH/MEDIUM 件数が **3 ラウンド連続で減らない**場合、観点設計が間違っているサインなのでオーケストレーターはユーザーに「観点を再設計するか、別アプローチに切り替えるか」を相談する

## 5. 観点ごとの追加チェック項目（観点ごとに必ずプロンプトに入れる）

### 観点 A 追加
- 章節アンカーの死活（`Grep` で md 内の `[...](./other.md#anchor)` を抜き出し、参照先見出しが実在するか）
- フェーズ番号の混在（複数計画間で同じ "Phase 1" が異なる意味で使われていないか）
- 旧用語の残存（前ラウンドで decided な廃止用語の grep カウント）

### 観点 B 追加
- 計画記述で関数名・モジュール名が実在のものと一致するか（`Grep` で対応コードを引いて確認）
- **行番号参照は陳腐化する**ので Finding として「シンボル名参照に置換」を推奨
- 移植元リポジトリ（他プロジェクト）の現状とのズレ（path で照合）

### 観点 C 追加
- SKILL の R1〜RN を網羅できているか（規約 ID をチェックリスト化）
- 互換不変条件（境界に漏出してはならない用語の集合）が機械検証できるよう lint タスク化されているか
- config キー名が docs に明示されているか（実装で骨抜きになる最大の温床）

### 観点 D 追加
- 各テストの観測点（pytest/cargo test ファイル名・実行コマンド）が明記されているか
- CI ゲート組込（`.github/workflows/`）が明記されているか — テスト存在と CI 組込は別物
- 不変条件 ID と test 関数名の対応表（`invariant-tests.md` 等）が起票されているか

## 6. 禁止事項

- LOW の Finding を理由にループ継続してはいけない（LOW は対応不要）
- Finding を「修正済み」とマークする前に、対象ファイルを `Read` で確認するか `Grep` で検証すること（サブエージェントの自己申告だけを信用しない）
- 計画文書に存在しない新機能・新フェーズ・新スコープを追加してはいけない
- SKILL の一次資料を計画文書側の記述で上書きしてはいけない（SKILL が正）
- 同一ファイルに複数 implementer を並列で書き込ませてはいけない（必ずファイル単位で分割）
- 統一方針を Step 2 で決めずに implementer を並列起動してはいけない（再分裂の主因）
- レビューサブエージェントが「収束」と返した瞬間にループ終了してはいけない。**最終ラウンドだけは grep ベースの機械検証を 1 セット必ず追加する**

## 7. 運用上の知見（実績ベース）

- **収束は線形には進まない**。修正で新しい記述が増えると新たな矛盾の余地が生まれる。HIGH/MEDIUM 件数は減るが、ゼロまで届くのに 5〜7 ラウンドかかることが多い
- **Finding 件数の推移**は監視せよ（ラウンド 1: 30〜40 件 / ラウンド 6: 0〜数件 が標準的な減衰）
- **ファイル単位並列・観点単位並列が鉄則**。Finding 単位の並列は競合事故が起きる
- **observation point（実行コマンド・テストファイル名）の明記**を観点 D で徹底すると、実装フェーズに入ってから「テストが書きづらい」事故が激減する
- **横串の波及修正（用語統一など）は 1 ラウンドで全ファイル同時修正**。複数ラウンドに分けない
- 中盤以降は**レビュー観点を合算**してエージェント数を絞ると体感速度が上がる（A+B+D 合算 / C 単独 など）
