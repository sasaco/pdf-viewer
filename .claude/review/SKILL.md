# Review

このスキルは、私が計画書をレビューするときの基準と手順をまとめたもの。
文章のきれいさよりも、**このまま実装者が動いたときに事故るかどうか** を優先して見る。

## 目的

- 実装前に、計画書の矛盾や危険な曖昧さを潰す
- 実装者が誤読しそうな箇所を先回りで見つける
- フェーズ順序、ゲート条件、受け入れ条件が実行可能かを確認する
- spec / architecture / open-questions との整合を取る

## いつ使うか

- `implementation-plan.md` をレビューするとき
- `spec.md` / `architecture.md` / `open-questions.md` の整合を確認するとき
- フェーズ着手前に「この計画で実装して大丈夫か」を見たいとき
- 実装中に、次のタスクへ進んでよいかを確認したいとき

## 最初に読むもの

最低でも以下を読む:

- 対象の計画書
- 同階層の `spec.md`
- 同階層の `architecture.md`
- 同階層の `open-questions.md`

計画書単体で閉じない。レビューは単一ファイルの校正ではなく、**文書群の整合確認**。

## レビューの進め方

### 1. 全体像を掴む

- 何をいつまでに実装する計画なのか
- どのフェーズがどの前提に依存するのか
- どこが source of truth なのか
- 未決事項がどこに残っているのか

### 2. 強い文言を先に拾う

最初に以下のような語を検索して、計画の骨格を掴む:

- `決定`
- `確定`
- `正本`
- `唯一`
- `禁止`
- `必須`
- `受け入れ`
- `TODO`
- `Phase`

例:

```bash
rg -n "決定|確定|正本|唯一|禁止|必須|受け入れ|TODO|Phase" docs/plan
```

### 3. 固有名詞を横断確認する

- enum 名
- DTO 名
- event 名
- key / field 名
- wire 値
- エラーコード
- フェーズ ID

1 箇所で見つけたら、その語の全出現箇所を追う。

### 4. findings first でまとめる

レビュー結果は要約より先に finding を出す。
優先順位は以下:

1. 実装を誤らせる矛盾
2. ゲートや受け入れ条件の破綻
3. 仕様未定義や命名不整合
4. テストや運用上の抜け

## 重点観点

### 1. 用語と wire 形式の一貫性

- 同じ概念が別表記になっていないか
- enum / schema / IPC 文字列が途中で変わっていないか
- UI 表示名と内部値を混同していないか

典型例:

- `D1` と `1d` が混在する
- `display_symbol` と `display_name_en` が混在する
- `VenueError` と旧 `EngineError` が併記される

### 2. 確定した方針を後段で覆していないか

- T0 で「実装位置を確定」と書いたのに、T4/T5 で再配置していないか
- open question が「決定済み」なのに、後段で未決のように扱っていないか
- 「source of truth」が複数あるように読めないか

特に見る語:

- `決定`
- `確定`
- `閉じる`
- `正本`
- `唯一`
- `実装位置`
- `責務`
- `source of truth`

### 3. フェーズ順序とゲート条件

- 「T5 着手前に必須」と「T1 着手前に必須」が混在していないか
- ブロッカーが未解決なのに受け入れ条件だけ先に進んでいないか
- リスク表と本文で依存関係がズレていないか

重要なのは、未解決ブロッカーの有無だけではなく、**どこまで進んでよいかの線引きが一貫しているか**。

### 4. 受け入れ条件の実装可能性

- 受け入れ条件が曖昧語で終わっていないか
- テスト名、入力、期待結果があるか
- ネガティブテストが必要なのに抜けていないか
- 壊れやすい性質に pin テストがあるか

危ない表現:

- `適切に処理する`
- `必要なら`
- `後で確認`
- `問題なければ`

### 5. 未定義フィールド・未定義データ経路

- 途中から新しい key / field / map 名が出ていないか
- 「送る」と書いてあるのに schema / DTO / event shape に載っていないものがないか
- 「別管理 map」と書いてあるのに、生成タイミングと所有者が不明ではないか

### 6. 実装者が誤読しやすい書き方

要注意:

- 肯定形と否定形が近接している
- 旧方針の否定が長文に埋もれている
- 同じ段落で `不要` と `必須` が混在する
- 例外ルールが脚注だけにある

レビューでは「正しいか」だけでなく、**急いで読む実装者がどう誤解するか** を見る。

### 7. Rust ↔ Python の両側性

フェーズが Rust と Python の両側にまたがるとき、片側だけ細かく書かれて反対側が薄くなりやすい。
**完了条件から逆算して、両側に対応する作業項目があるか** を確認する。

確認項目:

- IPC で送る DTO / event に、送信側と受信側の両方のタスクがあるか
- 受信した値をどの型に詰め直すかが書かれているか
- 値が落ちる経路がレビュー対象に入っているか

典型的な見落とし:

- Python 側 `list_tickers` だけ書かれていて Rust 側の受け皿が曖昧
- `new()` や normalize の経路で値が落ちるのに計画書に検証項目がない

### 8. ライフサイクル全体

「初回起動して 1 回動く」だけでは不十分。
再起動、再ログイン、状態切替、日跨ぎを含めて見る。

チェック項目:

- 再起動時に何が消え、何が残るか
- `VenueReady` などの再受信が idempotent か
- demo / prod 切替時に古い状態が残らないか
- JST 日跨ぎや session 期限切れ時の挙動があるか
- 長期稼働で単調増加する状態がないか

### 9. 複数入口の網羅

UI ガードだけでは止まらない。
他の入口から同じリクエストが飛ぶ可能性を確認する。

確認項目:

- 保存済み state 復元経路
- capabilities 適用前の race
- 直接 IPC
- 再接続後の自動再購読

各入口に対して、必要なら **backend 側で fail-safe + テスト pin** があるかを見る。

## 典型的な finding パターン

- `矛盾`: 同じ文書内、または関連文書間で方針が食い違う
- `未定義`: 後段で使う名前や field が前段で定義されていない
- `逆流`: 旧名称・旧 DTO・旧 error code が残っている
- `順序破綻`: 前提未確定のまま後続タスクが始められる
- `テスト不足`: positive case しかなく silent failure を防げない
- `責務曖昧`: Rust / Python / UI / backend のどこが正本か分からない

## 出力形式

各 finding には以下を含める:

- `path:line`
- 何が矛盾 / 不足 / 曖昧か
- なぜ危険か
- 可能なら、どちらに揃えるべきかの短い示唆

期待する書き方:

```text
- docs/plan/tachibana/implementation-plan.md:210
  T4 の受け入れ条件では `display_name_ja` を UI 表示に使う前提になっていますが、
  同文書の T2 では `display_symbol` を唯一の表示用キーとして確定しています。
  実装者がどちらを正とすべきか判断できず、Rust 側の保持構造が二重化する危険があります。
```

## 実務 tips

- 表、注記、リスク欄は本文とズレやすいので必ず見る
- 「完了根拠」「レビュー反映」ブロックは信用しすぎず、本文と照合する
- 1 件見つけたら、その用語の全出現箇所を追う
- 文体改善より先に、実装事故に直結する点を出す

## 修正提案を書くときの自己チェック

レビューで finding を出すだけでなく、自分で修正タスクを書くときは以下を必ずやる。
これを飛ばすと、次ラウンドで自分の修正が新しい矛盾になる。

### A. 既存確定方針への再 grep

修正タスクを書いた直後に、その本文に出てくる以下を grep して既存確定記述と一致するか確認する:

- 固有名詞
- 型名
- wire 値
- フィールド名
- フェーズ ID

確認例:

- wire 形式 (`1d` / `D1`)
- 責務の所在
- 確定済みフィールド名
- ゲート対象フェーズ

### B. 関数・クラス・ヘルパの存在確認

修正タスクで参照した関数やクラスが実コードに存在するか確認する。

- 関数・クラス・モジュールが実在するか
- 戻り値型が想定どおりか
- 呼ぼうとしている setter / getter / field が実在するか

### C. 概念書換時の波及確認

用語、型名、規約を書き換えたら、他文書にも旧表記が残っていないか grep する。

特に見る場所:

- `open-questions.md`
- `invariant-tests.md`
- `README`
- `SKILL.md`

## ひとことで言うと

良い計画書レビューは、文章をきれいにすることではなく、**実装者が間違っても不思議でない曖昧さや矛盾を先回りで潰すこと**。
